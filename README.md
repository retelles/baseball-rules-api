# Baseball Rules API

A FastAPI backend that manages baseball rulebook PDFs, user authentication, and usage analytics. Designed to power a mobile/web app where users can browse and download the latest rules documents.

---

## Features

- JWT-based auth (access + refresh tokens with rotation)
- Role-based access control (user / admin)
- PDF upload to Cloudflare R2 with version management
- Password reset via SendGrid email
- Usage event tracking
- Rate limiting on sensitive auth endpoints
- Security headers on all responses

---

## Setup

### 1. Clone and create a virtual environment

```bash
git clone <your-repo-url>
cd baseball-rules-api
python -m venv .venv
source .venv/bin/activate
```

### 2. Install dependencies

```bash
pip install -r requirements.txt
```

### 3. Configure environment variables

```bash
cp .env.example .env
```

Open `.env` and fill in every value (see [Environment Variables](#environment-variables) below).

### 4. Run database migrations

```bash
alembic upgrade head
```

### 5. Start the development server

```bash
uvicorn app.main:app --reload
```

The API will be available at `http://localhost:8000`. Interactive docs at `http://localhost:8000/docs`.

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `DATABASE_URL` | Yes | PostgreSQL connection string, e.g. `postgresql://user:pass@localhost:5432/baseball_rules` |
| `JWT_SECRET_KEY` | Yes | Long random string used to sign JWTs. Generate with `openssl rand -hex 32`. |
| `JWT_ALGORITHM` | No | JWT signing algorithm. Default: `HS256` |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | No | Access token lifetime in minutes. Default: `30` |
| `REFRESH_TOKEN_EXPIRE_DAYS` | No | Refresh token lifetime in days. Default: `7` |
| `SENDGRID_API_KEY` | Yes | SendGrid API key for transactional emails |
| `SENDGRID_FROM_EMAIL` | Yes | Verified sender address in your SendGrid account |
| `FRONTEND_URL` | Yes | Base URL of the frontend app (used in password-reset links), e.g. `https://yourapp.com` |
| `R2_ACCOUNT_ID` | Yes | Cloudflare account ID |
| `R2_ACCESS_KEY_ID` | Yes | R2 access key ID (from Cloudflare dashboard) |
| `R2_SECRET_ACCESS_KEY` | Yes | R2 secret access key |
| `R2_BUCKET_NAME` | Yes | Name of the R2 bucket where PDFs are stored |
| `ALLOWED_ORIGINS` | No | Comma-separated list of allowed CORS origins. Default: `*` (allow all). Set to your frontend URL(s) in production, e.g. `https://yourapp.com,https://admin.yourapp.com` |

## Creating the First Admin User

After running `alembic upgrade head`, connect to your database and run:

```sql
UPDATE users
SET role = 'admin'
WHERE email = 'you@yourdomain.com';
```

Or, if you want to create the admin directly without registering via the API:

```sql
-- Insert a user with a bcrypt-hashed password.
-- Generate the hash in Python first:
--   python -c "from passlib.context import CryptContext; print(CryptContext(schemes=['bcrypt']).hash('YourPassword1'))"

INSERT INTO users (id, email, hashed_password, role, is_active)
VALUES (
    gen_random_uuid(),
    'admin@yourdomain.com',
    '$2b$12$<your-bcrypt-hash>',
    'admin',
    true
);
```

## Deployment to Render

The repository includes a `render.yaml` blueprint that defines the web service and a managed PostgreSQL database.

### Steps

1. Push your code to a GitHub (or GitLab) repository.
2. In the [Render dashboard](https://dashboard.render.com), click **New > Blueprint** and connect your repository.
3. Render will detect `render.yaml` and create:
   - A **web service** running Uvicorn
   - A **PostgreSQL database** named `baseball-rules-db`
4. Set the following environment variables in the Render service's **Environment** tab (they are marked `sync: false` in `render.yaml` so they are never committed to source control):
   - `SENDGRID_API_KEY`
   - `SENDGRID_FROM_EMAIL`
   - `FRONTEND_URL`
   - `R2_ACCOUNT_ID`
   - `R2_ACCESS_KEY_ID`
   - `R2_SECRET_ACCESS_KEY`
   - `R2_BUCKET_NAME`
   - `ALLOWED_ORIGINS` — set to your production frontend URL(s)
5. `DATABASE_URL` is injected automatically from the linked database. `JWT_SECRET_KEY` is auto-generated by Render.
6. On each deploy, Render runs `alembic upgrade head` before starting the server (defined in `buildCommand` in `render.yaml`).
7. After the first deploy, create your admin user via the SQL snippet above using Render's **Shell** tab or a connected database client.

### Health check

Render uses `GET /health` to verify the service is running.
